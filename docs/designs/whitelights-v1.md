# Design: WhiteLights — Powerlifting Squat Depth Analyzer

Generated by /office-hours on 2026-04-14
Branch: main
Repo: vantage

Mode: Builder

## Problem Statement

The Rice powerlifting team needs a way to get instant, accurate squat depth feedback
during practice. Currently, depth judgment requires either a human coach watching from
the side or manual video review after the fact. Neither gives the lifter the immediate
pass/fail feedback they need to build depth confidence and consistency.

WhiteLights uses MediaPipe BlazePose pose estimation to analyze squat videos, detect
whether the hip crease passes below the top of the knee (IPF standard), and deliver
feedback in the language powerlifters already know: white lights (passed) or red lights
(no lift).

## What Makes This Cool

- The "white lights / red lights" UX is instantly legible to every powerlifter. No
  learning curve. You see the light and you know.
- The skeleton overlay on your own lift is genuinely surprising the first time you see
  it. You can watch your hip crease track past the knee threshold in real time.
- AI breakdown in meet-judge language, not fitness-app language: "Depth achieved at
  0:12. Bar path drifted 3cm forward. Back angle shifted 12 degrees at the hole.
  Estimated RPE: 7-8."

## Constraints

- Must run on a laptop with no GPU (CPU-only inference)
- First users are Rice powerlifting team members — they need to be able to run it
  without a complicated install
- Side-profile camera angle required for reliable depth judgment — this is a user
  responsibility, not a technical problem to solve
- Optional: Claude API key for AI breakdown (core depth call works without it)

## Premises

1. MediaPipe BlazePose over OpenPose — faster CPU inference, already validated for
   powerlifting-specific projects, no CUDA dependency.
2. Side-profile camera is required — hip crease vs. knee cap visibility depends on
   a true side view. Document this clearly in the README.
3. Video file input is the right MVP — iterate on the depth detection algorithm before
   fighting real-time latency.
4. Rice powerlifting team as first users — direct access, trust, immediate honest
   feedback.
5. Meet language (white lights / red lights) — the UX metaphor that makes this feel
   like a product, not a script.

## Approaches Considered

### Approach A: Python CLI
Python script + MediaPipe, terminal output only. Shippable in a day but no visual
output. Teammates have to trust a text result they can't verify.

### Approach B: Streamlit Web App (chosen)
Streamlit app: upload video, get skeleton overlay, depth call (with light visual),
frame-by-frame angle graph, and Claude-powered post-lift text breakdown. Runs locally,
shareable as a GitHub repo.

### Approach C: Session Analytics Dashboard
Full training session report: depth hit rate across attempts, fatigue curve, back angle
trends. Needs Approach B as the per-lift engine underneath — natural v2.

## Recommended Approach

**Approach B: Streamlit Web App**

### Architecture

```
whitelights/
  app.py              # Streamlit UI — upload, display, trigger analysis
  pose.py   # MediaPipe pose extraction + IPF depth threshold logic
  metrics.py          # Back angle + bar path calculations (consolidated, split later if needed)
  overlay.py          # OpenCV annotation: returns list of annotated frames (in-memory)
  analyzer.py         # Claude API call → 3-sentence post-lift breakdown (optional)
  requirements.txt
  README.md
```

### Key Technical Decisions

**Depth detection algorithm:**
- Use MediaPipe Pose landmarks: LEFT_HIP (23), LEFT_KNEE (25) for side-profile video.
  Note: BlazePose landmark 23 is the hip joint center, not the anatomical hip crease
  (the fold between thigh and torso). The crease is a few cm anterior. This approximation
  is acceptable for clear depth calls; borderline calls may require a calibration offset.
- For a side-profile camera, only the camera-side landmarks are reliable. Landmark
  selection rule: if `LEFT_HIP.visibility > 0.7`, use left-side landmarks (hip 23,
  knee 25); otherwise fall back to right-side (hip 24, knee 26). If neither side
  exceeds 0.7 confidence, return "indeterminate" — do not guess.
- IPF standard: hip crease must be BELOW the top of the knee cap.
- In 2D side view with y increasing downward: `hip_y > knee_y` = depth achieved.
- Find the "bottom of squat" frame: smooth the hip y-position across a 5-frame rolling
  average to suppress MediaPipe jitter, then find the frame where smoothed vertical
  velocity crosses zero (hip stops descending). Use wrist y-maximum as a fallback if
  no clean zero-crossing is detected.
- If no squat bottom is detected (video starts mid-descent), output: "Unable to determine
  depth — ensure the full descent is visible in the video."

**Depth call logic:**
- White lights: `hip_y > knee_y` for at least 3 consecutive frames within the ±5 frame
  window around the detected bottom frame
- Red lights: condition never met within that window
- "Close" indicator: hip within 5% of frame-height from knee threshold (borderline call)
- If bottom frame not detected: return "indeterminate" state, not a false red light

**Back angle:**
- Angle between shoulder landmark (LEFT_SHOULDER, 11), hip landmark (LEFT_HIP, 23),
  and vertical axis
- Track across all frames, report min/max/range and the frame where drift was greatest

**Bar path:**
- Track midpoint of LEFT_WRIST (15) and RIGHT_WRIST (16) across all frames
- Overlay as a traced line on the video frames
- Calculate lateral drift: max horizontal deviation from the starting wrist position,
  reported as % of frame width (not cm — no pixel-to-cm calibration available without
  a known reference object in frame)

**RPE estimation:** Moved to future work (v2). See below.

**Overlay rendering:**
- `overlay.py` processes all frames and returns a list of annotated BGR frames in memory
- `app.py` converts the frame list to a GIF (bottom 2 seconds) via Pillow and renders
  with `st.image()` — simpler than video encoding, avoids Streamlit `st.video` format
  issues
- Annotations per frame: MediaPipe skeleton, hip/knee landmark circles, horizontal
  threshold line at knee height, bar path trace, depth indicator flash at bottom frame

### Streamlit UI Flow

1. Upload video file (MP4, MOV). On ingest, run `cv2.CAP_PROP_ORIENTATION_AUTO` or
   read EXIF rotation metadata and rotate frames to upright before landmark extraction.
   iPhone MOV files commonly arrive rotated 90° at the sensor level — if not corrected,
   all landmark coordinates will be wrong. Cap input resolution to 720p; downsample
   if larger to keep GIF encoding tractable.
2. Processing spinner: "Analyzing your squat..."
3. Results display:
   - Big white lights / red lights graphic (the "whoa" moment)
   - Annotated GIF of the bottom 2 seconds (skeleton + depth indicator)
   - Depth achieved at: [timestamp]
   - Back angle range: X° to Y° (peak drift at [timestamp])
   - Bar path drift: X% of frame width lateral
   - [If ANTHROPIC_API_KEY env var present] AI breakdown: 3-sentence Claude analysis

### Claude API Prompt

```
You are a powerlifting coach reviewing a squat attempt. The athlete {hit/missed} depth.

Data from the lift:
- Depth: {hit/missed} (hip crease {was/was not} below knee cap)
- Closest depth frame: {timestamp}
- Back angle: ranged from {min}° to {max}°, peak drift at {timestamp}
- Bar path: {drift}% of frame width lateral drift
- Concentric phase: {frames} frames ({fast/moderate/slow})

Give a 3-sentence coaching breakdown in plain English. Be specific about what happened
and what to work on. Use powerlifting terminology. Do not use generic fitness language.
```

## Open Questions

1. **Camera angle tolerance:** How far off from true side-profile does the depth call
   break down? Need to test with video from slightly behind/in front of the lifter.
2. **Equipment occlusion:** Will MediaPipe reliably detect hip landmarks when the
   lifter wears a belt, knee sleeves, or squat suit?
3. **Video format:** MOV files from iPhone are common on the team. Need to verify
   OpenCV can read them without conversion.

**Resolved during design:**
- **Multi-person/spotter:** When multiple people are detected, use the largest bounding
  box as the primary subject (the lifter). Document in README: "For best results, film
  with only the lifter in frame or ensure the lifter is the closest/largest person."
- **RPE estimation:** Deferred to v2. Heuristic-based RPE without validation could
  confuse lifters. Build depth + back angle first, validate, then add RPE.

## Future Work (v2)

- **RPE estimation:** Bar concentric velocity + back angle drift as heuristic inputs to
  Claude. Label clearly as an estimate. Requires baseline calibration per lifter.
- **Real-time webcam mode:** Once algorithm is validated on video files, add live camera
  support with low-latency inference (target: <200ms from bottom frame to light display).
- **Session analytics:** Upload multiple lifts, get aggregate report (depth hit rate,
  back angle trend across attempts, fatigue curve). Approach C from this design session.

## Success Criteria

- Correctly calls depth (hit vs. no-hit) on 80%+ of lifts against a labeled test set of
  20 videos: 10 clear pass, 5 clear fail, 5 borderline — ground truth labeled by a human
  judge before running the algorithm
- Skeleton overlay renders correctly on uploaded video
- Claude breakdown generated within 5 seconds of video processing completion (run as
  async call while GIF renders; do not block the depth result on API response)
- At least 3 Rice powerlifting team members use it at practice and say it's useful
- README is clear enough that another college powerlifting team can clone and run it

## Distribution Plan

- GitHub repo: `rice-powerlifting/whitelights` (or personal fork)
- Install: `pip install -r requirements.txt && streamlit run app.py`
- Optional: Dockerfile for zero-dependency setup
- Claude API key: documented as optional in README, depth call works without it
- No hosted deployment needed for v1 — local run is fine for team use
